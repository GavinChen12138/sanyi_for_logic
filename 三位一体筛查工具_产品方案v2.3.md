# 浙江三位一体快速筛查工具 · 产品方案 v2.3

> 面向企业内部升学规划师，快速上线版本

---

## 一、项目概览

| 项目 | 内容 |
|------|------|
| 目标用户 | 企业内部升学规划师（< 1000人） |
| 登录方式 | 账号密码登录 |
| 角色体系 | `super_admin` / `admin` / `planner` 三级 |

## 二、角色与权限

### 角色说明

| 角色 | 说明 |
|------|------|
| `super_admin` | 超级管理员，拥有全部权限，包括高校数据管理 |
| `admin` | 普通管理员，可管理账号，不能操作高校数据 |
| `planner` | 规划师，只能使用成绩筛查功能 |

### 权限矩阵

| 功能 | super_admin | admin | planner |
|------|:-----------:|:-----:|:-------:|
| 成绩录入 & 筛查 | ✅ | ✅ | ✅ |
| 下载匹配结果 | ✅ | ✅ | ✅ |
| 查看账号列表 | ✅ | ✅ | ❌ |
| 创建账号 | ✅ | ✅ | ❌ |
| 重置他人密码 | ✅ | ✅ | ❌ |
| 启用 / 禁用账号 | ✅ | ✅ | ❌ |
| 升级 / 降级角色（仅 planner ↔ admin） | ✅ | ✅ | ❌ |
| 升级 / 降级为 super_admin | ✅ | ❌ | ❌ |
| 上传新高校 JSON（整体替换） | ✅ | ❌ | ❌ |
| 逐条编辑院校 / 专业组数据 | ✅ | ❌ | ❌ |
| 查看数据版本历史 & 回滚 | ✅ | ❌ | ❌ |

### 账号产生规则

```
部署时 → 运行初始化脚本 → 生成第一个 super_admin
super_admin → 可在后台创建任意角色账号
super_admin → 可把 admin 升级为 super_admin（或降级）
admin → 只能管理 planner，不能触碰 super_admin 账号
```

> **安全细节**：任何角色都不能修改、禁用、降级自己，防止出现无管理员的情况。

## 三、功能需求

### 3.1 登录模块

- 账号密码登录，后端校验返回 JWT（payload 含 role 字段）
- Token 存 `localStorage`，有效期 8 小时
- 过期或未登录自动跳回登录页
- 顶部导航按角色显示：
  - `planner` → 只有「筛查」
  - `admin` → 「筛查」+「账号管理」
  - `super_admin` → 「筛查」+「账号管理」+「数据管理」

### 3.2 账号管理模块（admin & super_admin）

账号列表展示：姓名、用户名、角色、状态（启用/禁用）、创建时间

操作：

- **创建账号**：弹窗填写姓名、用户名、初始密码、角色（admin 只能选 planner）
- **重置密码**：管理员设置新密码直接覆盖
- **切换角色**：admin 可操作 planner；super_admin 可操作全部角色
- **禁用 / 启用**：禁用后无法登录，数据保留

### 3.3 高校数据管理模块（仅 super_admin）

#### 3.3.1 数据版本列表

展示所有历史版本，字段：

| 字段 | 说明 |
|------|------|
| 版本号 | 自增，如 v1、v2、v3 |
| 上传时间 | 文件生效时间 |
| 上传人 | 操作者姓名 |
| 院校数量 | JSON 中的学校总数 |
| 专业组数量 | 所有 groups 总数 |
| 备注 | 上传时填写的更新说明 |
| 状态 | 「当前使用」/ 「历史版本」 |

操作：

- **回滚**：将任意历史版本设为当前使用版本，服务立即生效
- **下载**：可下载任意历史版本的 JSON 原文件备查

#### 3.3.2 上传新 JSON（整体替换）

操作流程：

1. super_admin 上传新的 `schools.json` 文件
2. 后端执行格式校验（见下方校验规则）
3. 校验通过 → 弹窗要求填写更新备注 → 确认后生效
4. 校验失败 → 显示具体错误信息，拒绝上传，旧版本继续生效

**校验规则**：

```
✅ 文件必须是合法 JSON
✅ 顶层必须是数组
✅ 每个学校必须有 id、name、groups 字段
✅ 每个 group 必须有 id、name、rules、entry_rule、required_subjects 字段
✅ rules 必须包含 A/B/C/D/E 五个键
✅ entry_rule 必须有 kind 和 score 字段
✅ required_subjects_mode 必须是 "and" 或 "or"
```

#### 3.3.3 逐条编辑

在页面上可以：

- 搜索 / 筛选院校（按名称、标签）
- 展开某院校查看所有专业组
- 点击专业组进入编辑弹窗，可修改以下字段：

| 可编辑字段 | 说明 |
|-----------|------|
| `preliminary_line` | 高考预审参考线 |
| `entry_rule.score` | 学考赋值分门槛 |
| `entry_rule_text` | 准入规则原文说明 |
| `rules` (A/B/C/D/E) | 各等级赋值分 |
| `required_subjects` | 选考科目要求 |
| `required_subjects_mode` | and / or |
| `last_year_composite_score` | 去年综合分参考 |
| `admission_count` | 招生人数 |
| `preliminary_line` | 高考预审线 |
| `description` | 专业描述 |

> **每次逐条编辑保存后，系统自动生成一个新版本**，版本备注自动记录为「逐条编辑：xxx大学-xxx专业组」，可在版本列表中查看和回滚。

### 3.4 成绩录入

规划师输入以下字段，全部必填：

| 字段 | 类型 | 说明 |
|------|------|------|
| 高考成绩 | 数字 | 裸分，0-750 |
| 选考科目 | 多选（6选3） | 物理、化学、生物、历史、地理、思想政治 |
| 语文学考 | 下拉 | A / B / C / D / E（五级制） |
| 数学学考 | 下拉 | 同上 |
| 英语学考 | 下拉 | 同上 |
| 选考3门学考 | 下拉 × 3 | 选中科目各填一个等级 |

> ⚠️ 学考等级为 A/B/C/D/E 五档，各档对应分值由各校自定义。

> **交互细节**：选考科目选定后，自动显示对应学考等级输入框，未选科目灰化隐藏。

### 3.5 匹配核心逻辑

**Step 1 · 选考科目过滤**

检查每个专业组的 `required_subjects` + `required_subjects_mode`：
- `mode = "and"`：学生必须全部选了这些科目
- `mode = "or"`：学生至少选了其中一门
- `required_subjects` 为空：无限制，全部通过

**Step 2 · 学考赋值分计算**

```
学生赋值分 = Σ（每门已填等级在该组 rules 中对应的分值）
```

**Step 3 · 学考准入规则判断**

| 类型 | 判断方式 | 标识 |
|------|----------|------|
| 数值型 | `entry_rule.score` 有具体数字，直接比较 | `rule_type: "score"` |
| 文字型 | `entry_rule_text` 为描述性文字，无法自动计算 | `rule_type: "text"` |

**Step 4 · 分级判断**

| 状态 | 判断条件 | 显示 |
|------|----------|------|
| ✅ 达线 | 学考赋值分 ≥ `entry_rule.score` | 绿色 |
| 🟡 接近 | 学考赋值分差距 ≤ 5分（可配置） | 黄色 |
| 🟡 接近 | 文字型规则，无法自动判断 | 黄色，标注「需人工核查」 |
| ❌ 不达标 | 差距超过容差，或选考科目不符 | 默认折叠 |

### 3.6 高考预审线展示

`preliminary_line` 仅作辅助参考，不参与达线判断：

- 卡片独立展示「高考预审线：xxx 分」及差值
- 超出为绿色，不足为橙色
- 不因预审线不足而降低匹配状态

### 3.7 结果展示 & 下载

每条结果卡片包含：

- 院校名称 + 专业组名称 + 标签
- **学考**：学生赋值分 / 要求分（或文字规则原文）/ 差值
- **高考预审线**（辅助）：学生分 / 预审线 / 差值
- 选考科目要求及满足状态
- 综合状态标签（达线 / 接近 / 需人工核查）
- 去年综合分参考 + 满分基准（100 或 750）
- 招生人数、专业描述

下载：**「下载 Excel」** / **「下载 PDF」**（均为前端生成，即时下载）

## 四、数据结构

### 4.1 高校数据 JSON 实际结构

```json
{
  "id": "school_22",
  "name": "温州医科大学",
  "tags": ["省重点一梯队", "medical"],
  "calcConfig": {
    "formula": "zhejiang_common",
    "weights": { "academic": 0.2, "interview": 0.2, "gaokao": 0.6 },
    "total_score": 100
  },
  "groups": [
    {
      "id": "school_22_g1",
      "name": "临床医学类",
      "rules": { "A": 10, "B": 9, "C": 7, "D": 4, "E": 0 },
      "required_subjects": ["物理", "化学"],
      "required_subjects_mode": "and",
      "entry_rule": { "kind": "score", "score": 96 },
      "entry_rule_text": "96.0",
      "preliminary_line": 0,
      "last_year_composite_score": 82.749,
      "admission_count": 115,
      "description": "包含专业：临床医学..."
    }
  ]
}
```

### 4.2 关键字段说明

| 字段 | 位置 | 说明 |
|------|------|------|
| `preliminary_line` | group | 高考预审参考线，仅辅助展示 |
| `entry_rule.score` | group | 学考赋值分准入门槛（数值型） |
| `entry_rule_text` | group | 准入规则原文（文字型时使用） |
| `rules` | group | 等级→赋值分对照，A/B/C/D/E 五级 |
| `total_score` | calcConfig | 满分基准：100 或 750 |
| `last_year_composite_score` | group | 去年录取综合分参考 |
| `required_subjects_mode` | group | `"and"` 全部满足；`"or"` 满足其一 |

### 4.3 数据库（SQLite）

```sql
-- 用户表
CREATE TABLE users (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  username    TEXT UNIQUE NOT NULL,
  password    TEXT NOT NULL,              -- bcrypt hash
  name        TEXT NOT NULL,
  role        TEXT NOT NULL DEFAULT 'planner',
              -- 'super_admin' | 'admin' | 'planner'
  is_active   INTEGER NOT NULL DEFAULT 1,
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 高校数据版本表
CREATE TABLE school_data_versions (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  version_no    INTEGER NOT NULL,         -- 自增版本号：1、2、3...
  file_path     TEXT NOT NULL,            -- 服务器上的 JSON 文件路径
  school_count  INTEGER NOT NULL,         -- 院校数量
  group_count   INTEGER NOT NULL,         -- 专业组数量
  note          TEXT,                     -- 上传时填写的备注
  uploaded_by   INTEGER NOT NULL,         -- 操作人 user.id
  uploaded_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
  is_active     INTEGER NOT NULL DEFAULT 0  -- 1=当前使用版本，同时只有1个为1
);
```

> **版本切换逻辑**：回滚时将目标版本 `is_active` 置 1，其余全部置 0，服务重新从对应 `file_path` 加载 JSON 进内存，无需重启进程。
